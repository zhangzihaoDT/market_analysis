# 告别“金鱼记忆”：智能数据分析 Agent 中的三层记忆架构设计与实践

在构建基于 LLM（如 DeepSeek）的智能数据分析 Agent 时，最经典的范式莫过于 **ReAct（Reason + Act）** 或 **Tool-Use Loop**（工具调用循环）。在这个基础架构下，Agent 接收用户问题 -> 思考思路 -> 调用 SQL 或 Python 工具 -> 根据报错修正 -> 最终输出结果。

然而，在实际业务场景落地时，这种“一次性”的循环系统暴露出一个致命缺陷：**Agent 就像一条只有 7 秒记忆的金鱼。**

今天，它可能因为不知道某张 CSV 表里的列名 `燃料类型 (组)` 带有空格和括号，写错了 SQL 而报错，经过两轮自我纠错（Self-Correction）终于查出了数据；明天，当用户再问一个类似的问题时，它依然会重蹈覆辙，先把列名写错，报错后再去修。

这种**“每次都在从零试错”**的模式，不仅极大地浪费了 Token 和执行时间（Latency），更让整个系统的可靠性停滞不前。

为了解决这个问题，我们在 SGM Agentic 项目中创新性地引入了 **“Evaluator + Strategy Memory + Strategy Injection” 三层学习架构**。本文将详细分享这一架构的设计思路、工作原理以及它带来的质变收益。

---

## 痛点剖析：传统 Loop 架构为什么不够用？

在传统的 `User -> Planner -> SQL Agent -> Analyst` 链路中：
1. **试错成本高昂**：业务数据往往是脏的（如大量交叉空值）、Schema 设计可能不符合常规 SQL 直觉（如列名带特殊字符、日期格式奇特）。Agent 每次遇到这些“坑”，都要靠报错堆栈来触发纠错。
2. **缺乏“隐性经验显性化”的机制**：优秀的业务分析师知道“算同环比必须用 `order_month` 而不能直接 `GROUP BY date`”，也知道“某张底表非常大，绝对不能 `SELECT *`”。但在纯 Prompt 驱动的 Agent 中，这些业务相关的 Anti-Pattern 很难被提前硬编码。
3. **陷入局部最优（Over-Exploration）**：有时候 Agent 查到了核心指标，但因为“过度探索”的惯性，非要再去查个区域分布、画个历史趋势，导致超时或超出最大循环轮数。

简而言之：**传统架构下，问题每次都是新问题；Agent 永远在重复交“学费”。**

---

## 破局之道：三层记忆（Memory）架构设计

我们并没有简单地把历史对话塞进 Prompt（那叫 Conversation History），而是设计了一个**提取和固化执行策略**的“认知框架”。

架构图如下：
```text
               （新增经验知识库）
        ┌──────────────────┐
        │ Strategy Memory  │ <── (3. 提取模式与反模式)
        └──────────────────┘
            ↓ (1. 注入)    ↑
User ─> Planner ─> SQL ─> Analyst
                   ↑ (2. 评估)
        ┌──────────────────┐
        │   Evaluator      │
        └──────────────────┘
```

这个架构由三个核心模块构成，分别对应**“监督、反思、指导”**的认知闭环：

### 1. Evaluator（质量评估器）：在后台默默打分
**目标：判断 Agent 这次干得好不好，并收集错题本。**

在每次工具（如 `execute_sql`）执行完毕后，我们在 Agent 的循环末尾挂载了一个轻量级的规则评估器。它不依赖大模型，纯靠代码逻辑评估执行质量：
- 如果 SQL 执行报错（如语法错误、找不到列），记录 `Execution Error`。
- 如果结果返回了 0 行（说明 SQL 语法对，但业务逻辑或过滤条件错了），记录 `empty_result`。
- 如果 SQL 中包含 `SELECT *`，记录 `low_precision`（低效查询）。

所有执行日志（无论成功与否）都会连同 Evaluator 给出的 `issues`，一起落盘到 DuckDB 构建的 `agent_logs` 表中。

### 2. Pattern Extractor（经验提取器）：从错题中学习
**目标：让隐性经验变成显性规则。**

我们引入了一个独立的**学习模式**（`python main.py --learn`）。
在夜间或业务低谷期触发此模式时，系统会强制开启大模型的**深度思考能力（Thinking/CoT）**，拉取 `agent_logs` 中那些带有 `issues`（如报错、结果为空）的近期执行记录。

大模型通过高算力深度反思这些“错题”，提取出结构化的经验，并 Upsert（更新/插入）到 `strategy_memory` 表中。
一个典型的经验（Pattern）结构如下：
```json
{
  "pattern": "查询包含特殊字符的字段名（如 燃料类型 (组)）",
  "strategy": "在 SQL 中必须严格使用双引号将字段名括起来",
  "anti_pattern": "直接在 SELECT 中书写带空格或括号的字段名"
}
```
**在这个环节，Thinking 模型扮演了“资深导师”的角色，它负责把底层踩过的坑，提炼成高维的策略。**

### 3. Strategy Injection（策略注入）：带着经验上路
**目标：让 Agent 从“零思考”升级为“基于经验决策”。**

在下一次用户提问时，系统会拦截用户的 Query，并使用 `jieba` 分词进行轻量级的 Jaccard 相似度匹配（Strategy Retriever），从 `strategy_memory` 中检索出 Top-3 最相关的历史经验。

这些高度相关的经验会被动态组装进 Planner 的 System Prompt 中。例如：
> **历史经验（高相关）：**
> - **问题模式**: 统计新能源渗透率
> - **推荐策略**: 计算出核心指标（渗透率 %）后，立即停止探索，输出最终答案。
> - **避免做法**: 查完渗透率后继续去查各区域分布或历史同比。

**此时，接收到任务的 Agent 不再是一个“愣头青”，而是一个手里拿着《该业务避坑指南》的老手。**

---

## 核心收益：为什么这个架构是质变？

引入这套三层 Memory 架构后，系统的表现发生了肉眼可见的蜕变：

### 1. Zero-Shot（一次性做对）成功率大幅提升
之前，遇到带有脏数据或不规则命名的表，Agent 必然要经历“报错 -> 修正 -> 成功”的 3-4 轮循环。现在，由于它提前在 Prompt 中看到了 `anti_pattern` 警告，它能在**第一轮生成 SQL 时就完美规避这些错误**。

### 2. Token 消耗与 Latency（延迟）双降
少走弯路意味着减少了工具调用的轮数（Loop Rounds）。原本需要 6 轮交互（包含多次带有冗长 traceback 的报错上下文返回）才能得出的答案，现在通常 2-3 轮就能锁定胜局。响应速度翻倍，API 成本锐减。

### 3. 彻底根治“过度探索（Goal Drifting）”
大模型经常会因为“好奇心”或惯性，在算出目标数据后继续调用工具进行画蛇添足的分析（比如用户只问总销量，它非要按省份再 group by 一次），导致严重超时。通过将“强制终止条件”作为最高优先级红线写入 Prompt，并结合经验注入，Agent 学会了“见好就收”。

### 4. 真正意义上的“越用越聪明”
传统的 Agent 系统能力上限取决于开发者的 Prompt 功底。而这套架构赋予了系统**自我进化的能力**。新入职的业务人员可能会问出奇葩的业务逻辑导致 Agent 崩溃，但只要执行一次 `--learn`，第二天系统就能完美处理同类需求。

## 总结

我们加的并不是一个简单的“历史对话缓存”，而是一个模拟人类专家**“实践 -> 评估 -> 总结归纳 -> 指导下一次实践”**的完整认知框架。

在算力有限、Token 昂贵、且要求低延迟的商业数据分析场景中：
- **日常对话模式**：关闭高昂的 Thinking，依赖轻量级的经验检索，享受快速、低成本的推理。
- **夜间学习模式**：火力全开（启用 Thinking），深度榨取大模型的推理能力，将经验浓缩提炼固化。

这套“Evaluator + Strategy Memory + Injection”的设计，让智能 Agent 真正摆脱了“金鱼记忆”，向着能沉淀领域知识的“业务专家”迈出了坚实的一步。
